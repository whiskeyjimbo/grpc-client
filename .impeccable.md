## Design Context

### Users
Backend engineers debugging and testing gRPC services in development/staging, and QA engineers verifying happy/sad path behavior. Used actively during coding sessions: live service exploration, quick call execution, environment variable tweaking, and soon scenario test suite verification. Users are technical, expect precision and speed, and tolerate density in exchange for power.

### Brand Personality
Precise, powerful, exploratory. The tool should feel like a well-machined instrument — not flashy, not corporate, not playful. Think: a calibrated oscilloscope or a well-tuned terminal emulator. Confidence without decoration.

### Aesthetic Direction
Dark theme (engineers using this in low-light dev environments, often alongside running servers in terminal windows). NOT: the standard AI dark palette (navy + cyan + purple). Should feel more industrial/utilitarian — dense information layout, tight typography, high contrast where it matters. Anti-reference: anything that looks like a SaaS marketing page gone dark.

### Design Principles
1. **Execution is the primary action.** Everything else is in service of getting to Execute and reading the response.
2. **Show the work.** Variables should resolve visibly. Commands (grpcurl/curl) should be readable, not just copyable.
3. **Density with legibility.** Engineers work in tight windows. Information should be packed but scannable.
4. **Fail loudly, succeed quietly.** Errors deserve attention. Success states should not interrupt flow.
5. **Power users first, discoverability second.** Keyboard shortcuts, not wizards.
